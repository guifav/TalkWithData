# Use case roadmap

Talk With Data currently supports one product concept: the internal BI
distribution portal with row-level security described in the
[internal BI portal setup guide](INTERNAL-BI-PORTAL-SETUP.md). One deployment
serves one organization; the organization's own team signs in with the
company Google Workspace domain and talks with the company's own data.

This document records the use-case directions beyond that concept, so
contributors can pick a front and design discussions have a stable reference.
Like [ROADMAP.md](../ROADMAP.md), nothing here is a commitment or a date.
Security, maintainability, and maintainer capacity come first.

## How to contribute to a front

1. Read [CONTRIBUTING.md](../CONTRIBUTING.md) and
   [GOVERNANCE.md](../GOVERNANCE.md).
2. Open a GitHub issue describing the slice you want to take, referencing the
   section below. Fronts that touch authentication or token boundaries need a
   design discussion before code. Report suspected vulnerabilities through
   [SECURITY.md](../SECURITY.md), never in public issues.
3. Keep slices small and independently reviewable.

## 1. Selective distribution to external clients

**Concept.** A company shares governed data slices with its own clients: each
client signs in, or receives a link, and sees only the data that belongs to
them.

**What exists today.** Embed tokens allow sharing a single dashboard
externally: random 7-day tokens, read-only, bound to one dashboard, minted by
authenticated users with view access. Sign-in itself is locked to one Google
Workspace domain by design, so external identities cannot authenticate. Row
scope keys off the authenticated viewer, so it does not apply to embed
viewers.

**Missing pieces.**

- An external identity model: multi-domain allowlist, email invitations, or
  federation, plus the Firestore rules and server checks that follow from it.
- An external viewer role with a deliberately restricted surface.
- Embed hardening: configurable expiry, revocation, service-issued tokens,
  and per-token usage visibility.
- A tenancy decision: whether one deployment can serve several audiences, or
  whether one isolated deployment per audience remains the recommendation.

**Entry points.** This is the largest conceptual change in the list, and it
moves the trust boundary of the whole application. Start with a design
proposal issue, not a pull request.

## 2. Versioned report hub workflows

**Concept.** Teams that produce recurring HTML reports, generated by AI,
notebooks, or other tooling, publish them to the hub as the distribution
channel of record.

**What exists today.** Single HTML and ZIP package upload, search across
extracted text, categories, shared folders, per-dashboard versions capped at
10 with FIFO cleanup for single-file dashboards (ZIP package replacement
keeps no version history), thumbnails, and view analytics.

**Missing pieces.**

- Publishing conventions: draft and published states, review before
  team-wide visibility.
- Version history for ZIP package replacements.
- Retention and archival policies beyond the 10-version cap.
- Bulk import and export tooling.
- Notifications when a followed dashboard is updated.

**Entry points.** Mostly self-contained application work; a good first front
for contributors familiar with Next.js and Firestore.

## 3. Conversational self-service analytics at scale

**Concept.** Analysts and non-technical staff answer their own questions
through chat, and the organization treats the chat as a governed analytics
surface rather than a demo.

**What exists today.** Dataset chat over governed CSV sources with per-viewer
row scope; an AI dashboard builder with a per-dashboard structured database;
MCP integration for live external tools; a governed prompt registry; per-user
model selection across several providers.

**Missing pieces.**

- Governed source adapters beyond CSV buckets, such as warehouse and database
  connectors, preserving the same grant and row-scope contract.
- Cost governance: per-user or per-team quotas and usage reporting for AI
  spend.
- Saved and shareable analyses.
- Defined behavior for datasets larger than the in-memory engine comfortably
  holds.

**Entry points.** The source adapter contract under
`app/src/lib/data-sources/` is the deepest and most valuable front. Each new
adapter deserves its own design issue.

## 4. Embedded analytics in existing portals

**Concept.** Dashboards from the hub render inside an intranet, a wiki, or an
internal product through iframes, without the application shell.

**What exists today.** Token-based embed views with a strict sandbox,
read-only data access, and 7-day expiry.

**Missing pieces.**

- Configurable token lifetime and revocation.
- A service API for minting and rotating embed tokens from automation.
- Theming for embedded contexts.
- Per-token usage analytics.

**Entry points.** Well-bounded server and UI work, but it changes token
semantics; align on the contract in an issue first.

## 5. BI license cost reduction playbook

**Concept.** Organizations keep report authors on their existing BI platform
and move read-only consumers to the hub, reducing per-viewer licensing while
keeping governance.

**What exists today.** Everything in the
[internal BI portal setup guide](INTERNAL-BI-PORTAL-SETUP.md): governed
extracts, row scope, chat, and dashboards.

**Missing pieces.**

- Reference export-job recipes. Core docs stay vendor-agnostic; concrete
  recipes could live under an `examples/` tree.
- A migration playbook: choosing datasets, mapping audiences to owner
  columns, and validating parity with the source BI platform.
- Load and performance guidance for viewer-scale deployments.

**Entry points.** A documentation-first front, well suited to contributors
who operate the tool in production.
